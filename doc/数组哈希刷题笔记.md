# 数组哈希刷题笔记

## 模块核心

数组哈希题的核心不是 HashMap API，而是设计映射关系：

```text
key 表示什么？
value 表示什么？
为什么这个 key 能把同类数据归到一起？
```

常见用途：

- 查找补数；
- 记录出现位置；
- 统计频次；
- 分组；
- 前缀状态计数。

## Day 11：索引映射与频次签名

- 实刷日期：2026-06-18
- 题目：1. 两数之和；49. 字母异位词分组

### 1. 两数之和

这题是哈希表的基础模型：

```text
当前值：nums[i]
目标补数：target - nums[i]
```

遍历时先查询补数是否已经出现，再把当前值放入 map。这样可以避免同一个元素被重复使用。

映射设计：

```text
Map<数值, 下标>
```

关键点：

- 先查再放；
- value 存下标，不是存数字本身；
- 如果有重复数字，后出现的下标可以覆盖前面的下标，但本题找到答案后直接返回，影响不大。

### 49. 字母异位词分组

异位词的判断标准是字符频次相同。可以为每个字符串生成一个“频次签名”，把签名相同的字符串放到同一个列表里。

映射设计：

```text
Map<频次签名, 字符串列表>
```

Java 注意点：

```java
String key = Arrays.toString(count);
```

不要写：

```java
count.toString();
```

因为数组的 `toString()` 不是内容字符串，而是对象标识。

### 今日沉淀

这两题分别代表哈希表的两类基本用法：

```text
1. 两数之和：用哈希表做 O(1) 查找。
2. 字母异位词分组：用哈希表做分类聚合。
```

数组哈希模块后续重点是：

- key 的唯一性；
- value 的语义；
- 是否需要处理重复值；
- 是否可以用频次、排序、前缀和、余数等结构作为 key。

## Day 12：TopK 与前缀和

- 实刷日期：2026-06-21
- 题目：347. 前 K 个高频元素；692. 前 K 个高频单词；560. 和为 K 的子数组；523. 连续的子数组和

### 347 / 692：TopK 高频问题

TopK 高频问题通常分两步：

```text
1. HashMap 统计频次
2. 堆 / 桶排序 / 排序 取前 K
```

今天的关键修正是：PriorityQueue 不会因为外部 `Map` 的 value 改变而自动重排。因此应当先完整统计频次，再把元素放入堆。

347 只需要按频次排序；692 还需要处理频次相同时的字典序：

```text
频次高的在前
频次相同，字典序小的在前
```

### 560：和为 K 的子数组

这题是前缀和计数：

```text
prefix[j] - prefix[i] = k
=> prefix[i] = prefix[j] - k
```

因此遍历到当前前缀和 `prefix` 时，需要知道之前出现过多少次 `prefix - k`。

由于数组里可能有负数，不能用滑动窗口。

推荐顺序是：

```text
先 ans += count(prefix - k)
再 count(prefix)++
```

这样不需要对 `k == 0` 做额外修正。

### 523：连续的子数组和

这题是前缀和余数：

```text
(prefix[j] - prefix[i]) % k == 0
=> prefix[j] % k == prefix[i] % k
```

所以只要两个前缀和对 k 的余数相同，中间这段子数组的和就可以被 k 整除。

关键点：

- `mp.put(0, -1)` 用于处理从下标 0 开始的子数组；
- Map 中保存某个余数第一次出现的位置；
- 必须检查子数组长度至少为 2。

### 今日关键词

- TopK：先统计，再建堆；
- PriorityQueue 不会监听外部 Map 变化；
- 560：前缀和出现次数；
- 523：前缀和同余 + 最早下标；
- 有负数时优先考虑前缀和，不要套滑动窗口。
