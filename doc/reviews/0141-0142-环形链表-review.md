# LeetCode 141 / 142 环形链表 Review

## 141. 环形链表

### 评价

这题代码是标准快慢指针写法，逻辑正确，边界条件也完整。

优点：

- `head == null` 直接返回，避免空链表场景。
- 循环条件使用 `fast != null && fast.next != null`，能保证 `fast.next.next` 安全。
- `fast` 每次走两步，`slow` 每次走一步，相遇则说明有环。

可以优化的点：

- `head == null` 的特判不是必须，因为 `while (fast != null && fast.next != null)` 已经能覆盖；保留也没问题，可读性更直接。
- Java 风格上建议后续逐渐统一空格，例如 `if (head == null)`、`while (fast != null && fast.next != null)`。

核心思维：

- 如果无环，`fast` 会先走到 `null`。
- 如果有环，`fast` 进入环后会不断缩短和 `slow` 的相对距离，最终相遇。

## 142. 环形链表 II

### 评价

这题代码结构清晰，分成两阶段：

1. 快慢指针找相遇点；
2. 一个指针回到 `head`，另一个留在相遇点，同步走找到入口。

优点：

- 先判断 `head == null`，边界安全。
- 第一阶段循环条件安全：`fast != null && fast.next != null`。
- 找不到环时通过 `fast == null || fast.next == null` 返回 `null`，逻辑正确。
- 第二阶段 `fast = head` 后和 `slow` 同速前进，最终返回入口节点，模板正确。
- 你在代码末尾写了数学推导，这是好习惯，说明不是在死记模板。

可以优化的点：

- 推导里的变量名可以更标准化，比如用：
  - `a`：head 到入口距离；
  - `b`：入口到相遇点距离；
  - `c`：相遇点到入口距离。
- 现在注释里的 `len1 / len2 / m / n` 能表达思路，但面试时建议换成 `a,b,c`，更容易讲清楚。
- `beginTime` 和 `endTime` 都是 `10:40`，如果是实际记录没问题；如果之后想统计用时，建议精确到分钟并保证提交时间更新。

## 环入口推导的标准表达

设：

```text
a = head 到环入口距离
b = 环入口到第一次相遇点距离
c = 第一次相遇点回到环入口距离
```

第一次相遇时：

```text
slow = a + b
fast = a + b + k(b + c)
fast = 2 * slow
```

所以：

```text
2(a + b) = a + b + k(b + c)
a + b = k(b + c)
a = (k - 1)(b + c) + c
```

这说明：

- 从 `head` 出发走 `a` 步会到入口；
- 从相遇点出发走 `a` 步，也会先走 `c` 到入口，再绕若干圈回到入口。

所以第二阶段两个指针同步走，第一次相遇点就是环入口。
