## 148. 排序链表

### 代码评价

代码使用递归归并排序，整体正确：

1. base case：空链表或单节点链表天然有序；
2. 快慢指针找中点；
3. 断开左右两段；
4. 递归排序左右子链表；
5. merge 两个有序链表。

### 面试表达要点

为什么链表排序选择归并：

- 要达到 `O(n log n)`，常见选择是快排或归并；
- 数组上快排常见，因为支持随机访问、partition 方便；
- 链表不支持随机访问，随机 pivot 和原地 partition 都不自然；
- 归并排序只依赖顺序遍历和合并两个有序链表，复杂度稳定，适合链表。

为什么必须断链：

```java
ListNode second = slow.next;
slow.next = null;
```

如果不断链，左子链表仍然是全集，不会真正缩小，递归无法收敛，最终可能栈溢出。

base case：

```java
if (head == null || head.next == null) return head;
```

含义是：长度为 0 或 1 的链表已经有序，不需要继续拆分。

### 可优化点

- `merge` 中如果希望保持稳定性，可以使用 `<=`，相等时优先取左链表节点。
- 递归版空间复杂度是 `O(log n)` 调用栈；如果严格要求 `O(1)` 额外空间，可以写自底向上归并。

## 23. 合并 K 个升序链表

### 代码评价

代码使用小根堆，整体正确：

- 初始化时把每条非空链表的头节点放入堆；
- 每次弹出当前最小节点接到答案链表后面；
- 如果该节点还有 `next`，把 `next` 放回堆；
- 堆中最多维护 k 个候选节点。

### 面试表达要点

为什么可以用小根堆：

```text
每次需要从 k 条链表当前头节点中选出最小节点。
```

堆里维护的是：

```text
每条非空链表当前还没有合并部分的头节点。
```

时间复杂度：

```text
每个节点最多入堆一次、出堆一次；堆大小最多 k；所以总时间 O(N log k)。
```

空间复杂度：

```text
堆最多存 k 个节点，所以 O(k)。
```

和 21 的关系：

```text
21 是两路归并；23 是 k 路归并。小根堆把“从 k 个候选中选最小”的过程优化到 O(log k)。
```

### 可优化点

当前比较器：

```java
(a, b) -> a.val - b.val
```

理论上存在整数溢出风险。更稳的写法：

```java
(a, b) -> Integer.compare(a.val, b.val)
```

这不是稳定性问题，而是整数减法可能溢出的问题。
