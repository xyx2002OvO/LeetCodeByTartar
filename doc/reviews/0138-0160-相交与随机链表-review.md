# LeetCode 160 / 138 链表节点身份 Review

## 160. 相交链表

### 代码评价

最终代码使用双指针换头法，思路正确：

- `curA` 走完 A 后切到 B；
- `curB` 走完 B 后切到 A；
- 两个指针走过的虚拟路径长度相同，最终会在交点或 `null` 相遇。

核心抽象是：

```text
curA: A + B
curB: B + A
```

这里的相交判断必须是节点身份相同：

```java
curA == curB
```

不是节点值相等。

### 今日修正点

一开始的写法把“切换到另一条链表头”和“继续走 next”放在同一轮里，会跳过对方链表头节点。正确理解是：

```text
null -> headB 本身就是本轮移动结果，不能再 .next
```

因此推荐写法是：

```java
curA = (curA == null) ? headB : curA.next;
curB = (curB == null) ? headA : curB.next;
```

## 138. 随机链表的复制

### 代码评价

最终代码使用递归 DFS + HashMap 记忆化，思路正确。带 `random` 指针的链表可以抽象成一个特殊图：

```text
next 是一条边
random 也是一条边
```

所以复制过程本质是图拷贝。`Map<Node, Node>` 的作用是维护：

```text
原节点 -> 复制节点
```

并保证同一个原节点只创建一次。

### 今日修正点

第一版尝试在遍历主链时提前创建 `random` 节点，这会导致同一个原节点可能被复制两次。正确原则是：

```text
一个原节点只能对应一个复制节点
```

可以用两遍 HashMap，也可以用递归 DFS + HashMap。最终递归版能自然处理 random 指回前面节点、random 成环等情况。

### 可优化点

当前代码中：

```java
public Map<Node, Node> map = new HashMap<>();
```

刷题可用，但从工程风格看建议改成 `private`，或者把 `map` 放入 helper 参数中，避免同一个 `Solution` 对象多次调用时状态残留。

## 今日沉淀

今天两题的共同核心是“节点身份”：

- 160 判断两条链表是否相交，看的是两个指针是否指向同一个节点对象；
- 138 复制复杂链表时，HashMap 的 key 必须是原节点对象，而不是节点值。

一句话总结：

```text
链表高级题不要只看 val，要看节点身份和指针关系。
```
