# 链表刷题笔记

## 模块核心

链表题的重点是维护指针语义，而不是单纯写循环。每次动指针前先确认：

- 当前节点是谁；
- 前驱节点是谁；
- 结果链表尾部是谁；
- 移动后不变量是否仍然成立。

## 通用技巧

### dummy 节点

删除、拼接、构造新链表时优先使用 dummy。它可以统一处理头节点，避免单独讨论“head 被删除”或“结果链表第一个节点怎么接”。

### 快慢指针

快慢指针类题目要先定义距离关系。例如删除倒数第 N 个节点，本质不是直接找目标节点，而是让 `slow` 停在目标节点的前驱位置。

### 拼接链表

合并两个有序链表时，主循环只负责处理两个链表都非空时的比较逻辑。某一边为空后，剩余部分可以一次性挂接，不需要继续逐节点处理。

### 数字链表

做链表加法题时，第一步判断数字存储方向：

- 低位在前：直接从头节点加；
- 高位在前：反转链表或使用栈，把问题转成从低位开始处理。

## 当前编码风格要求

- `dummy`：虚拟头节点；
- `cur`：当前结果链尾；
- `fast` / `slow`：快慢指针；
- `next`：链表临时后继节点；
- `carry`：进位，不要命名成 `next`；
- 提交前一个 Java 文件只保留一个 `class Solution`。

## Day 3：反转链表

### 206. 反转链表

反转链表要先建立指针不变量：`prev` 是已经反转好的部分，`cur` 是当前待处理节点，`next` 用来保存原链表后续，防止断链。

递归版的关键是理解函数语义：`reverse(head)` 返回以 `head` 为头的链表反转后的新头。回溯时通过 `head.next.next = head` 把当前节点接到尾部，再用 `head.next = null` 断尾，避免形成环。

### 92. 反转链表 II

局部反转本质是“固定区间前驱，然后不断把区间内后续节点头插到前驱后面”。

关键指针：

- `pre`：反转区间前一个节点；
- `cur`：原区间第一个节点，反转过程中会变成区间尾部；
- `next`：每轮从 `cur.next` 摘下并头插的节点。

循环次数是 `right - left`，因为第一个节点本来就在区间里，只需要把后面的节点逐个前插。

### 关于 `end` 边界思维

92 题里使用 `end` 作为反转区间的右边界，本质上是从 206「反转整条链表」自然扩展出来的：

- 反转整条链表：`while (cur != null)`；
- 反转局部区间：`while (cur != end)`。

这是一种“半开区间”思维：反转 `[start, end)`，即从 `start` 开始，到 `end` 之前停止。它比单纯记忆 92 的头插次数更通用，后续可以迁移到 25「K 个一组翻转链表」。

需要注意的是，`end` 的定位要尽量写得直观。推荐先定位 `pre` 到 `left` 前一个节点，再从 `pre` 或 `start` 出发找到 `right` 后一个节点。不要让 `end` 先走一段、再和 `pre` 同步移动，否则虽然数学上能推对，但读代码的人需要额外推导。

可以把这个思路总结为：

```text
整链表反转：反转 [head, null)
局部链表反转：反转 [start, end)
K 组链表反转：反转 [groupStart, groupEnd)
```

## Day 4：环形链表

- 结束时间：2026-06-11 10:55

### 141. 环形链表

判断是否有环使用快慢指针：`fast` 每次走两步，`slow` 每次走一步。如果无环，`fast` 会先到 `null`；如果有环，两个指针进入环后相对距离会不断变化，最终相遇。

安全循环条件是：

```java
while (fast != null && fast.next != null)
```

这样才能安全执行 `fast = fast.next.next`。

### 142. 环形链表 II

这题在 141 的基础上多做一步：找到第一次相遇点后，让一个指针回到 `head`，另一个留在相遇点，两个指针每次都走一步，再次相遇的位置就是环入口。

标准推导可以记成：

```text
a = head 到入口距离
b = 入口到相遇点距离
c = 相遇点到入口距离

slow = a + b
fast = a + b + k(b + c)
fast = 2 * slow
=> a = (k - 1)(b + c) + c
```

所以从 `head` 和第一次相遇点同时出发，最终会在入口相遇。

代码里记录 `beginTime/endTime` 是好习惯，后续可以用于统计每类题的耗时。环形链表这类题建议把数学推导写在代码注释或 review 里，避免只背“相遇后一个回 head”。

## Day 5：相交链表与随机链表复制

- 开始时间：2026-06-12 10:26

### 160. 相交链表

相交链表的核心是“双指针换头”，逻辑上让两个指针分别走虚拟路径：

```text
A + B
B + A
```

如果两条链表相交，两个指针会在第一个公共节点相遇；如果不相交，两个指针会同时走到 `null`。

今天的关键修正是：切换链表头节点本身就是本轮移动结果，不能切到另一条链表头之后再 `.next`，否则会跳过对方链表的第一个节点。

### 138. 随机链表的复制

这题的本质不是普通链表复制，而是特殊图拷贝：`next` 和 `random` 都是边。

最重要的原则是：

```text
一个原节点只能对应一个复制节点。
```

因此需要 `Map<Node, Node>` 维护原节点到新节点的映射。HashMap 两遍法更直观，递归 DFS + HashMap 更接近图拷贝模型。

### 今日关键词

- 节点身份，而不是节点值；
- 虚拟拼接，而不是真实改链；
- old node 到 new node 的唯一映射；
- random 链表可以用图拷贝思维处理。

## Day 6：回文链表与重排链表

### 234. 回文链表

回文链表可以拆成三个动作：

```text
找中点 -> 反转后半段 -> 前后同步比较
```

今天的写法使用 dummy 出发的快慢指针，让 `slow` 停在前半段尾部，然后 `mid = slow.next` 作为后半段起点。断开 `slow.next = null` 后反转后半段，再和前半段逐个比较。

这个题的关键不是“回文”本身，而是把链表转成两个方向一致的比较过程。

### 143. 重排链表

重排链表是 234 的升级版，前半部分完全一样：

```text
找中点 -> 断链 -> 反转后半段
```

区别在于最后一步不再是比较，而是交错合并：

```text
L0 -> L1 -> L2
R0 -> R1 -> R2
合并成 L0 -> R0 -> L1 -> R1 -> L2 -> R2
```

交错合并时必须先保存后半段的 `next`，否则插入当前节点后会丢失后续链表。

### 今日关键词

- 组合模板：找中点 + 断链 + 反转 + 比较/合并；
- 234 关注比较；
- 143 关注交错合并；
- 如果题目要求保持原链表，234 比较后要恢复后半段。

## Day 7：K 个一组翻转链表与旋转链表

- 实刷日期：2026-06-14

### 61. 旋转链表

旋转链表的关键是先把 `k` 对链表长度取模：

```text
k = k % len
```

如果 `k == 0`，旋转后链表不变。否则新尾节点是原链表第 `len - k` 个节点，新头节点是 `newTail.next`。

可以选择两种实现方式：

1. 先让原尾节点连回 `head` 成环，再找到新尾并断开；
2. 先找到新尾，断开后再把原尾接回 `head`。

今天代码使用的是第二种，本质和成环法等价。

### 25. K 个一组翻转链表

这题是 92 局部反转的升级版。核心仍然是头插法：固定当前组前驱 `pre`，把组内后续节点不断摘出来插到 `pre` 后面。

关键变量：

- `pre`：当前组的前驱；
- `tail`：用于检查当前组是否够 k 个节点；
- `end`：当前组后一个节点，是反转停止边界；
- `newHead`：当前组原来的第一个节点，反转后变成当前组尾节点；
- `cur`：当前要被摘出来头插的节点。

今日调试点是漏写了：

```java
cur = newHead.next;
```

这行必须在每一组开始反转前重新执行，因为每一组的 `cur` 都应该从当前组第二个节点开始。如果漏掉，就会沿用上一轮旧指针，导致头插过程错位。

### 今日关键词

- 61：长度取模、新尾、新头、断环/接环；
- 25：先验组长、半开区间 `[groupHead, end)`、头插反转、组间拼接；
- 复杂链表反转题要特别关注“每一轮开始时指针是否重新定位”。

## Day 8：多路归并与排序链表

- 实刷日期：2026-06-15

### 148. 排序链表

链表排序选择归并排序的原因：链表不支持随机访问，快速排序里的随机 pivot 和原地 partition 都不如数组自然；归并排序只依赖顺序遍历和合并两个有序链表，复杂度稳定。

递归归并排序的结构是：

```text
base case -> 找中点 -> 断链 -> 递归排序左右两段 -> merge
```

base case 是递归终止条件：空链表或单节点链表已经有序，直接返回。

断链很关键：如果不断开 `slow.next`，左子链表仍然是原全集，递归规模不会缩小，最终可能栈溢出。

### 23. 合并 K 个升序链表

这题是 21「合并两个有序链表」的 k 路扩展。小根堆维护的是每条非空链表当前还未合并部分的头节点。

复杂度：

```text
N = 所有节点总数
k = 链表数量
时间复杂度 O(N log k)
空间复杂度 O(k)
```

每个节点最多入堆一次、出堆一次；堆大小最多是 k。

### 今日关键词

- two-way merge 到 k-way merge；
- 链表归并排序；
- base case = 递归出口；
- 断链保证递归规模缩小；
- 比较器用 `Integer.compare` 避免整数溢出。

### 今日评分

综合评分：8.6 / 10。

优势：模型识别和主代码正确性较好，能把 21 的 merge 模板迁移到 23 和 148。

改进点：比较器细节、代码格式、递归版空间复杂度说明需要继续稳定。

### 707. 设计链表

这题本质是把链表基本操作封装成 API。今天使用的是双向链表 + dummy head/tail + size。

关键边界：

- `get(index)`：`index < 0 || index >= size` 返回 -1；
- `addAtIndex(index, val)`：`index == size` 合法，表示尾插；`index > size` 不插入；
- `deleteAtIndex(index)`：删除前必须确认 `index` 在 `[0, size)`。

这题和 LRU 的底层双向链表操作是同源的，后续可以统一抽象为：

```text
addAfter(prev, node)
remove(node)
```

### 430. 扁平化多级双向链表

这题是 DFS + 双向链表拼接。遇到 `child` 时，需要：

```text
1. 保存原 next
2. flatten child
3. child 链接到 cur.next
4. 找 child 链尾
5. child 链尾接回原 next
6. cur.child = null
```

今天代码逻辑正确，关键指针都维护到了：

- child 接入时更新 `prev`；
- 原 next 接回时更新 `prev`；
- 扁平化后清空 `child`。

需要注意的是，当前写法每次通过 `getTail` 找尾节点，极端情况下可能重复扫描。更优雅的面试写法是 DFS helper 返回扁平化后的尾节点。

### 今日关键词

- 707：API 边界、dummy head/tail、size 维护；
- 430：保存原 next、接 child、找尾、接回、清 child；
- 设计型链表题要优先抽象基础操作，减少重复指针代码。

这题是哈希表的入门模型：


遍历时先查补数是否已经出现，再把当前值放入 map。这样可以避免同一个元素被使用两次。

核心抽象：

```text

```

#### 49. 字母异位词分组

异位词的本质是字符频次相同。可以用长度为 26 的数组统计每个单词的字符频次，再把频次数组转成字符串作为哈希 key。

Java 注意点：

```java
Arrays.toString(count)
```

```java

```

因为数组的 `toString()` 不是内容字符串，而是对象标识。

## 数组哈希 Day 12：TopK 与前缀和

TopK 高频题先统计频次，再用堆/桶排序取结果。注意 Java 的 PriorityQueue 不会随着外部 Map 的 value 变化而自动重排，所以比较器依赖 Map 时，Map 在堆使用期间应保持稳定。

前缀和题要先识别目标：

- 560 是“前缀和差值等于 k”，所以 Map 存前缀和出现次数；
- 523 是“前缀和差值能被 k 整除”，所以 Map 存余数第一次出现的位置。

有负数时滑动窗口通常失效，前缀和 + 哈希表更稳。
